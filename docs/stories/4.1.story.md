## Story

**Como** integrante do squad de desenvolvimento do Living Lab (papel `@dev`, com a DDL dos contained users delegada a `@data-engineer` e o provisionamento de identidade/RBAC a `@devops`, sob o gate `@architect`),
**eu quero** substituir toda autenticação por connection string no data-plane (SQL, Service Bus, ACR, Log Analytics) por Managed Identity + Entra ID, mover os segredos irredutíveis (chaves de LLM e `GATEWAY_SHARED_SECRET`) para o Key Vault lidos via KV reference resolvida pela MI, e criar contained users de menor-privilégio no Azure SQL (McpServer = **`db_datareader`-only**),
**para que** nenhum segredo de serviço viva em texto puro em App Setting/repo/log (ADE-009 Invariante 2), a regra de ouro do chatbot ("nunca escreve direto no banco", ADE-008 Inv 1) passe a valer **por garantia de RBAC do próprio banco** — não só por código — e a Story 4.2 tenha uma fonte segura de onde ler o `GATEWAY_SHARED_SECRET` que fecha o P0 do bypass das Functions.

2. **AC-2 — Dependência `Azure.Identity` adicionada aos dois projetos que hoje NÃO a têm:** `Fifa2026.V2.Functions.csproj` e `Fifa2026.V2.McpServer.csproj` passam a referenciar `Azure.Identity` **versão `1.13.1`** — a **MESMA** já pinada em `Fifa2026.V2.FlowEvents.csproj` (verificado; é o único projeto que hoje a usa, para `DefaultAzureCredential` via `Azure.Monitor.Query`).    *(Fonte: verificado `Fifa2026.V2.FlowEvents.csproj` — `Azure.Identity` 1.13.1; ADE-009 §Verificação "padrão MI já existe".)*

3. **AC-3 — `PurchaseRepository.cs` (Functions) e `FifaQueryRepository.cs` (McpServer) autenticam no SQL via Managed Identity, não mais connection string com senha:** ambos os construtores hoje leem `configuration["SqlConnectionString"]` e lançam `InvalidOperationException` se ausente (verificado: `PurchaseRepository.cs` linhas 25-30; `FifaQueryRepository.cs` linhas 22-28) — **esse guard fail-closed é PRESERVADO intocado**. Muda apenas a **FORMA** do valor de `SqlConnectionString`: de `Server=...;User Id=...;Password=...` para uma connection string com **`Authentication=Active Directory Managed Identity`** (formato nativamente suportado por `Microsoft.Data.SqlClient` **5.2.2**, já pinada em ambos os projetos — **sem upgrade de pacote**) e, se a MI for **User-Assigned**, `User Id=<client-id-da-MI>` adicional (padrão documentado Microsoft — nenhuma API inventada). Nenhuma outra linha de `new SqlConnection(_connectionString)` (`PurchaseRepository.cs` linha 80; `FifaQueryRepository.cs` uso equivalente) muda — é literalmente uma troca de STRING, não de mecanismo de chamada.
   *(Fonte: verificado `PurchaseRepository.cs` linhas 22-31, 80; `FifaQueryRepository.cs` linhas 17-28; ADE-009 Inv 2 tabela — "Azure SQL... token AAD (Authentication=Active Directory Managed Identity)".)*

4. **AC-4 — Pré-requisito de infraestrutura descoberto nesta story: Azure AD admin no SQL Server.** Verificado em `infra/modules/sql-database.bicep` (linhas 31-41, recurso `sqlServer`): **não existe** nenhum recurso `Microsoft.Sql/servers/administrators` (Azure AD admin) hoje — só `administratorLogin`/`administratorLoginPassword` (SQL auth, humano). Autenticação AAD/MI (incluindo os contained users `FROM EXTERNAL PROVIDER` da Task 3) **não funciona** num Azure SQL Server sem pelo menos um Azure AD admin configurado no servidor. Esta AC adiciona esse recurso ao `sql-database.bicep` (parametrizado — o `objectId`/`login` do AD admin é um parâmetro, não hardcoded, ADE-003 "URL/valor nunca hardcoded"). **`[AUTO-DECISION]`** — achado técnico não coberto explicitamente pela ADE-009 §Verificação; sinalizado para o `@architect` confirmar no gate desta story (não é uma nova invariante, é um pré-requisito operacional da Inv 2 já decidida).
   *(Fonte: verificado `infra/modules/sql-database.bicep` linhas 31-41 — ausência confirmada por leitura completa do arquivo; conhecimento de plataforma Azure SQL — AAD admin é pré-requisito documentado da Microsoft para autenticação Entra em Azure SQL.)*

5. **AC-5 — `administratorLogin`/`administratorLoginPassword` (SQL auth) do servidor NÃO são removidos nesta story:** permanecem como credencial de administração humana/instrutor (fora do data-plane de aplicação).
